# シェル芸 LLM 評価ルーブリック

## 1. 目的

この文書は、ShellGeiAI が扱う「シェル芸」の定義と、LLM が生成・評価する候補コマンドの評価基準をまとめる。

ShellGeiAI は、単に正答するコマンドだけを探すのではなく、シェルらしさ、発想、安全性、再現性、鑑賞性まで含めて候補を比較する。そのために、評価を `judge`、`score`、`selector` の3段に分ける。

- `judge`: 実行可能性、正確性、stderr、危険コマンド、制約違反を見る gate
- `score`: 正答候補に対して、簡潔性、シェルらしさ、発想、可読性、堅牢性、鑑賞性を採点する quality layer
- `selector`: `judge` を通過した候補から、評価モードに合うものを選ぶ

正答性は前提条件である。正しく動かない候補を、短さ、発想、鑑賞性で救済しない。

## 2. ShellGeiAI におけるシェル芸

ShellGeiAI におけるシェル芸は、CLI による問題解決の俳句である。

短いだけではなく、道具選び、標準入出力の流れ、問題の見方の転換、そして落ちがある。テキストやストリームとして与えられた問題を、小さな道具のパイプラインで変換し、答えに到達する表現として扱う。

```text
問題
  ↓
テキスト / ストリーム
  ↓
小さな道具のパイプライン
  ↓
答え
```

シェル芸は「長いスクリプトの1行化」ではない。シェル、パイプ、標準入出力、既存コマンドを使うからこそ、問題が美しく解ける形を探すことである。

中心の問いは次の一文に集約できる。

> その問題を、シェルの道具立てだからこそ美しく解いているか。

## 3. 許容範囲

ShellGeiAI では、シェル芸の定義を広めに取る。

`grep`、`sed`、`awk`、`sort`、`uniq`、`cut`、`paste`、`tr`、`xargs`、`find` などの Unix コマンドを中心に評価する。ただし、言語ワンライナーを避けること自体は目的ではない。`perl -ple`、`perl -a`、`node -pe`、短い `python -c` などを、標準入出力につながる小さな道具としてうまく使っているなら、それはシェル芸として高く評価できる。

たとえば、厳密な JSON ではない JSON-like な入力を `node -pe` で自然に扱う、`perl -ple` や `perl -a` でレコード処理を短く表す、Python の標準ライブラリや広く知られたライブラリを短く読み込んで複雑な処理を一撃で表す、といった使い方は評価対象になる。

重要なのは、どの言語を起動したかだけではない。標準入出力を使い、問題をテキスト処理やストリーム処理として捉え、言語処理系も含めた小さな道具の組み合わせで解いているかを見る。

一方で、ほぼ通常プログラムを別言語で書き、シェルが起動役にしかなっていない候補は、正答していてもシェルらしさや鑑賞性を低く評価する。評価が高いのは、言語ワンライナーがパイプラインの流れ、問題の見立て、道具選びの必然性に噛み合っている場合である。

## 4. 正答性 Gate

正答性がない候補は 0 点とする。

次の候補は、原則として採点対象外または 0 点にする。

- 構文エラーで実行できない
- 問題の出力仕様を満たさない
- 入力を無視して固定出力する
- サンプルケースだけをハードコードする
- 問題文の制約に違反する
- ファイル、環境、ネットワークに危険な影響を与える
- タイムアウト、大量 stderr、非決定的な出力などにより、実行結果が不安定

芸術性は加点要素である。正答性 gate を通過した候補同士を比べるときに、発想、一撃感、鑑賞性を評価する。

## 5. 評価モード

評価モードは、用途に応じて次の3つに分ける。

### 5.1 `simple`

単純さ優先。勉強会や即答の用途を想定する。

正答性 gate を通った候補の中で、道具選びが素直で、無駄な処理が少なく、説明なしでも使いやすいものを高く評価する。実行速度も見るが、主眼は速さそのものではなく、少ない道具と短い流れで正しく答えに到達することに置く。

### 5.2 `artistry`

芸術優先。シェル芸としての鑑賞性を重視する。

正答性 gate を通った候補の中で、見方の転換、道具選びの妙、一撃感、落ちの美しさを強く評価する。短いだけのコードゴルフではなく、問題がシェルの形へうまく変換されているかを見る。

### 5.3 `robustness`

堅牢性優先。実務利用や再現性を重視する。

正答性 gate を通った候補の中で、入力の揺れ、境界条件、環境差、安全性、読みやすさを強く評価する。やや長くても、壊れにくく説明しやすい解法を選ぶ。

## 6. 共通採点項目

正答性 gate を通過した候補に対して、次の6項目を採点する。

| 項目 | 評価すること |
|---|---|
| 簡潔性 | 短く、無駄な処理が少ないか |
| シェルらしさ | パイプ、標準入出力、Unix コマンド、シェル機能を自然に使っているか |
| 発想 | 問題の見方を変え、整形、並べ替え、集約、集合演算などとして捉え直しているか |
| 可読性 | 人間が処理の流れを追えるか |
| 堅牢性 | 入力の揺れ、境界条件、環境差、安全性に強いか |
| 鑑賞性 | 見た人が「その手があったか」と感じるか |

### 6.1 簡潔性

短さと無駄のなさを評価する。ただし、短ければ必ず高得点ではない。

見るべき点:

- コマンドやパイプ段数が問題に対して妥当か
- 不要な `cat`、一時ファイル、重複処理がないか
- 短縮のために正確性や可読性を壊していないか

### 6.2 シェルらしさ

シェル、Unix コマンド、言語ワンライナー、標準入出力を自然に使っているかを評価する。

見るべき点:

- `grep`、`sed`、`awk`、`sort`、`uniq`、`cut`、`paste` などを適切に組み合わせているか
- `perl -ple`、`perl -a`、`node -pe`、短い `python -c` などを、標準入出力上の小さな道具として活かしているか
- パイプラインの各段階に役割があるか
- シェルが単なる別言語プログラムの起動役に留まらず、道具同士をつなぐ場として機能しているか

### 6.3 発想

問題を別の形に捉え直しているかを評価する。

見るべき点:

- 逐次処理ではなく、整形、並べ替え、集約、集合演算として捉えているか
- 入力制約やデータ構造を見抜いているか
- 見た人が「その手があったか」と感じるか

### 6.4 可読性

人間が処理の流れを追えるかを評価する。

見るべき点:

- パイプラインの各段階の意図が推測できるか
- 正規表現や `awk` の処理が説明可能か
- コードゴルフ目的の難読化になっていないか

### 6.5 堅牢性

入力の揺れ、境界条件、環境差、安全性に対する強さを評価する。

見るべき点:

- 空入力、空白、タブ、日本語、記号を含む入力に対応できるか
- 最終行に改行がない場合でも破綻しないか
- 大量データで極端に遅くならないか
- GNU/BSD 差、macOS/Linux 差に依存しすぎていないか
- `rm`、`eval`、危険な `xargs`、未クォート変数などを危険に使っていないか

### 6.6 鑑賞性

シェル芸として見たときの面白さ、美しさ、一撃感を評価する。

見るべき点:

- 道具選びに必然性と意外性があるか
- 問題が美しく変換されているか
- 説明を聞いた人が納得・感心できるか
- 単なる短縮ではなく「芸」として成立しているか

## 7. モード別配点

100 点満点で評価する。ただし、採点対象は正答性 gate を通過した候補だけである。

| 項目 | `simple` | `artistry` | `robustness` |
|---|---:|---:|---:|
| 簡潔性 | 25 | 15 | 10 |
| シェルらしさ | 20 | 20 | 15 |
| 発想 | 15 | 25 | 10 |
| 可読性 | 10 | 10 | 20 |
| 堅牢性 | 20 | 10 | 35 |
| 鑑賞性 | 10 | 20 | 10 |
| 合計 | 100 | 100 | 100 |

### 7.1 `simple` の評価傾向

`simple` では、短く、素直で、余計な処理が少ない候補を高く評価する。シェル芸としての面白さは加点するが、複雑な見立てや技巧よりも、少ない道具で問題を自然に解く候補を選ぶ。

### 7.2 `artistry` の評価傾向

`artistry` では、問題の見方の転換、道具選びの妙、一撃感を高く評価する。正答性 gate を通過している限り、多少の説明の難しさよりも、シェル芸としての鑑賞性を重視する。

### 7.3 `robustness` の評価傾向

`robustness` では、壊れにくさ、環境差への強さ、安全性、説明可能性を高く評価する。少し長くても、再現しやすく、利用者が安心して読める候補を選ぶ。

## 8. 大幅減点または失格条件

次の条件に該当する候補は、0 点、採点対象外、または大幅減点にする。

| 条件 | 推奨処理 |
|---|---|
| 実行不能、構文エラー | 0 点 |
| 出力仕様を満たさない | 0 点 |
| 入力を無視して固定出力する | 0 点 |
| サンプルケースだけをハードコードする | 0 点 |
| ファイルや環境を破壊する可能性が高い | 0 点、または失格 |
| ネットワークアクセスや外部サービスに依存する | 制約違反として 0 点、または大幅減点 |
| 問題文で禁止されたコマンドを使う | 0 点、または失格 |
| ほぼ他言語の通常プログラムで、シェルが起動役だけ | シェルらしさ、簡潔性、鑑賞性を大幅減点 |
| 言語ワンライナーがパイプラインや標準入出力と噛み合っていない | シェルらしさと鑑賞性を減点 |
| 不要な stderr、警告、環境依存がある | 堅牢性と可読性を減点 |

## 9. コードゴルフとの違い

コードゴルフでは、短さが主要な評価軸になることが多い。

シェル芸では、短さに加えて次を評価する。

- シェルらしい道具の組み合わせ
- パイプラインの流れ
- 問題の見方の転換
- 制約下での変換美
- 実用性との距離感
- 鑑賞性

そのため、最短解が必ずしも最高のシェル芸とは限らない。短くても、正答性がなければ 0 点である。正答していても、道具選びや流れに必然性がなければ高く評価しない。

## 10. 評価出力フォーマット

LLM が評価する場合は、次の形式で出力する。

````markdown
# シェル芸評価

## Gate

- 結果: passed / failed
- 理由: ...

## 総合点

XX / 100

## 評価モード

simple / artistry / robustness

## 各項目

| 項目 | 点数 | 短評 |
|---|---:|---|
| 簡潔性 | XX / YY | ... |
| シェルらしさ | XX / YY | ... |
| 発想 | XX / YY | ... |
| 可読性 | XX / YY | ... |
| 堅牢性 | XX / YY | ... |
| 鑑賞性 | XX / YY | ... |

## 良い点

- ...

## 気になる点

- ...

## シェル芸としての評価

...

## 改善案

```sh
...
```

## 評価の確度

- 実行確認: あり / なし
- 注意点: ...
````

Gate が `failed` の場合、総合点は 0 点にする。失敗候補に対して、芸術性や発想の加点を行わない。

## 11. 評価プロンプト例

LLM に評価させる場合は、次のようなプロンプトを使う。

```text
あなたはシェル芸の評価者です。

以下の問題と候補コマンドを、ShellGeiAI のシェル芸 LLM 評価ルーブリックに従って評価してください。

まず gate として、実行可能性、正確性、出力形式、stderr、危険コマンド、制約違反を確認してください。
gate を通過しない場合、総合点は 0 点にしてください。短さ、発想、鑑賞性による救済はしないでください。

gate を通過した場合だけ、指定された評価モードに従って 100 点満点で採点してください。

評価モード:
simple / artistry / robustness

評価では、正答性だけでなく、簡潔性、シェルらしさ、発想、可読性、堅牢性、鑑賞性を分けて見てください。
主観的な点数には必ず理由を添えてください。
実行確認ができない場合は、静的解析に基づく推定であることを明記してください。

問題文:
...

入力仕様:
...

期待出力:
...

候補コマンド:
...
```

## 12. 最終的な評価思想

ShellGeiAI が向き合うシェル芸は、正答だけでも、短さだけでもない。

正しく動くかは工学である。短く無駄がないかは技法である。見てうなるかは芸である。

```text
正しく動くか
  = 工学

短く無駄がないか
  = 技法

見てうなるか
  = 芸
```

ただし、この順序は重要である。正答性がなければ 0 点であり、芸術性は正答候補に対する加点要素として扱う。

LLM は評価時に、常に次の問いへ戻る。

> その問題を、シェルの道具立てだからこそ美しく解いているか。
