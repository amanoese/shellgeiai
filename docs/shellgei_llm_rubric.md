# シェル芸 LLM評価ルーブリック

## 実装との対応

ShellGeiAI では、このルーブリックを次の 2 層で使う。

- `judge`: 正答性、実行可能性、stderr などの失格条件を判定する gate
- `shellgei score`: 簡潔性、シェルらしさ、発想、可読性、堅牢性、鑑賞性を独立に採点する quality layer
- `selector`: `best-score-wins` の場合に `judge` と `shellgei score` を合わせて最終候補を選ぶ

現在の score mode は `standard` / `competition` / `practical` / `appreciation` を受け付ける。現段階の実装では `standard` を既定とし、結果表示とログにも mode を残す。

## 1. 目的

この文書は、LLMが生成または評価するシェル芸について、評価基準を一貫させるためのルーブリックである。

シェル芸は、単に「シェルで書かれた短いワンライナー」ではない。UNIX的な道具の組み合わせ、問題の見方の転換、簡潔性、実用性、可読性、そして鑑賞性を含む表現である。

LLMによる評価では、実行可能性や正確性といった客観的評価に加え、「シェルらしさ」「発想の飛躍」「芸術性」のような主観的評価も扱う必要がある。

---

## 2. シェル芸の定義

### 2.1 基本定義

シェル芸とは、シェル、UNIXコマンド、標準入出力、パイプライン、テキストストリームなどを活用し、与えられた問題を簡潔かつ巧妙に解く表現である。

より短く定義すると、次のように表せる。

> シェル芸とは、CLIによる問題解決の俳句である。  
> 短いだけではなく、道具選び、流れ、発想、落ちがある。

また、評価上は次のように定義する。

> シェル芸とは、シェルの道具立てだからこそ成立する、制約下の変換美である。

### 2.2 シェル芸に含まれる要素

シェル芸には、主に次の要素が含まれる。

| 要素 | 説明 |
|---|---|
| 正確性 | 問題の仕様に対して正しい出力を返すこと |
| 簡潔性 | 無駄が少なく、必要最小限の記述であること |
| シェルらしさ | パイプ、標準入出力、既存コマンドの組み合わせを活かしていること |
| 発想の飛躍 | 問題を別の見方に変換して解いていること |
| 可読性 | 人間が読んで処理の流れを追えること |
| 堅牢性 | 境界条件、入力揺れ、環境差にある程度耐えること |
| 芸術性 | 見た人が「なるほど」「その手があったか」と感じること |

### 2.3 シェル芸ではないもの

以下は、必ずしもシェル芸として高く評価しない。

| 例 | 理由 |
|---|---|
| 単に長いスクリプトを1行に詰め込んだだけのもの | ワンライナーではあるが、芸ではない |
| ほぼすべてを `python -c` や `perl -e` に逃がしたもの | シェルの道具立てを活かしていない |
| 動かない、または仕様を満たさないもの | 芸術性以前に問題解決として成立していない |
| 危険なコマンドを不用意に使うもの | AI生成物として安全性に問題がある |
| 入力例だけに過剰適合したもの | 一般的な解法ではなく、テストハックに近い |

## 3. 評価の全体構造

LLMは、シェル芸を次の3層で評価する。

```text
正しく動くか        = 工学
短く無駄がないか    = 技法
見てうなるか        = 芸
```

総合点は100点満点とする。

| 大分類 | 配点 |
|---|---:|
| A. 正確性・実行可能性 | 30点 |
| B. 簡潔性・無駄のなさ | 15点 |
| C. シェルらしさ | 15点 |
| D. 発想の飛躍 | 15点 |
| E. 可読性・説明可能性 | 10点 |
| F. 堅牢性・安全性 | 10点 |
| G. 芸術性・鑑賞性 | 5点 |
| **合計** | **100点** |

---

## 4. 評価前提

LLMが評価する際は、次の情報を前提として受け取ることが望ましい。

```text
問題文:
入力仕様:
期待される出力:
制約:
提出されたシェル芸:
実行環境:
使用可能コマンド:
評価モード:
```

評価モードは、必要に応じて次のいずれかを指定する。

| モード | 重視するもの |
|---|---|
| 競技モード | 短さ、発想、一撃感 |
| 実務モード | 正確性、堅牢性、可読性、保守性 |
| 鑑賞モード | 発想、意外性、美しさ |
| 標準モード | 全項目をバランスよく評価 |

指定がない場合は、標準モードで評価する。

---

## 5. 評価項目と得点基準

## A. 正確性・実行可能性: 30点

### 評価観点

提出されたコマンドが、問題の仕様を満たして正しく動作するかを評価する。

### LLMが見るべき点

- 構文エラーがないか
- 指定された入力形式に対応しているか
- 期待される出力と一致するか
- 余計な出力をしないか
- 標準エラーに不要な警告を出さないか
- 問題文の制約を満たしているか
- 特定のサンプルだけに過剰適合していないか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 実行不能、構文エラー、または問題と無関係 |
| 1〜5点 | ごく一部のサンプルのみ正しい |
| 6〜10点 | 基本的な入力では一部正しいが、多くのケースで失敗する |
| 11〜20点 | 主要ケースは正しいが、境界ケースや仕様の一部で失敗する |
| 21〜27点 | ほぼ正しい。軽微な仕様漏れや環境依存がある |
| 28〜30点 | 仕様を正確に満たし、余計な出力もない |

### 減点例

| 問題 | 減点目安 |
|---|---:|
| 構文エラー | -30 |
| 出力形式が異なる | -5〜-15 |
| サンプルだけに依存 | -10〜-25 |
| 入力を無視して固定出力 | -20〜-30 |
| 標準エラーに不要な警告 | -1〜-5 |

---

## B. 簡潔性・無駄のなさ: 15点

### 評価観点

コードが短く、冗長な処理がなく、必要最小限の道具で書かれているかを評価する。

ただし、短ければ必ず高得点ではない。短さと無駄のなさを分けて評価する。

### LLMが見るべき点

- 文字数、バイト数が過度に長くないか
- 不要な `cat`、`grep`、`sort`、一時ファイルがないか
- 同じ処理を複数回行っていないか
- コマンド数やパイプ段数が問題に対して妥当か
- 短縮のために可読性や正確性を壊していないか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 不必要に長大、またはほぼ通常スクリプトを1行化しただけ |
| 1〜3点 | 冗長な処理が多く、シェル芸として締まりがない |
| 4〜7点 | 動くが、明らかに不要なコマンドや重複処理がある |
| 8〜11点 | 比較的簡潔で、無駄は少ない |
| 12〜14点 | 短く、処理の重複も少ない |
| 15点 | 非常に簡潔で、ほぼ削る余地がない |

### 減点例

| 問題 | 減点目安 |
|---|---:|
| 不要な `cat` | -1〜-2 |
| 不要な中間ファイル | -2〜-5 |
| 明らかな二重処理 | -3〜-7 |
| 長いだけの埋め込みスクリプト | -5〜-15 |
| 短縮しすぎて正確性を損なう | 正確性でも大きく減点 |

---

## C. シェルらしさ: 15点

### 評価観点

シェル、UNIXコマンド、標準入出力、パイプラインの思想を活かしているかを評価する。

### LLMが見るべき点

- データが標準入力から標準出力へ自然に流れているか
- 小さなコマンドを適切に組み合わせているか
- `grep`, `sed`, `awk`, `sort`, `uniq`, `cut`, `paste`, `tr`, `xargs`, `join`, `comm` などを適材適所で使っているか
- シェル展開、プロセス置換、リダイレクトを効果的に使っているか
- 何でも他言語に逃げていないか
- シェルで書く必然性があるか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 実質的にシェルではない。ほぼ他言語の実行ラッパー |
| 1〜3点 | シェル上で動くが、シェルの特性をほとんど使っていない |
| 4〜7点 | 一部シェルらしいが、道具選びが不自然 |
| 8〜11点 | 標準的なシェル処理として自然 |
| 12〜14点 | UNIX的な道具の合成がよく効いている |
| 15点 | シェルでなければこの美しさは出ない、という必然性がある |

### 減点例

| 問題 | 減点目安 |
|---|---:|
| ほぼ `python -c` のみ | -8〜-15 |
| 一時ファイルに強く依存 | -2〜-6 |
| 標準入出力を活かしていない | -3〜-8 |
| 道具選びが不自然 | -2〜-5 |

---

## D. 発想の飛躍: 15点

### 評価観点

問題を別の見方に変換し、意外だが妥当な方法で解いているかを評価する。

### LLMが見るべき点

- 問題の構造を見抜いているか
- データの並び替え、転置、反転、集約などで問題を簡単にしているか
- コマンドの通常用途をうまくずらして使っているか
- 数学的性質や入力制約を活かしているか
- 典型解とは異なるが、納得感のある解法か
- 見た人が「その手があったか」と感じるか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 発想上の工夫がない、または力技のみ |
| 1〜3点 | ほぼ素直な逐次処理で、意外性は少ない |
| 4〜7点 | 一部に工夫がある |
| 8〜11点 | 問題の見方をうまく変えている |
| 12〜14点 | 非自明な変換や道具選びが鮮やか |
| 15点 | 解法の視点そのものが美しく、強い驚きがある |

### 加点例

| 工夫 | 加点目安 |
|---|---:|
| ソートや集合演算として問題を捉え直す | +2〜+5 |
| 行列転置・反転・正規化などで一気に簡単化 | +2〜+5 |
| 通常用途と異なるコマンド活用 | +2〜+5 |
| 問題制約を見抜いて処理を大幅削減 | +3〜+7 |

---

## E. 可読性・説明可能性: 10点

### 評価観点

人間が読んだときに、処理の流れや意図を追えるかを評価する。

競技的なシェル芸では多少の難読性は許容するが、LLM評価では「説明可能な難しさ」と「単なる読みにくさ」を区別する。

### LLMが見るべき点

- パイプラインの各段階の役割が推測できるか
- 正規表現や `awk` の処理が過度に難解でないか
- 短縮のためだけの不可解な記法が多すぎないか
- 説明すれば納得できる構造か
- 実務で使う場合に分解・保守できるか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | ほぼ解読不能 |
| 1〜2点 | 動作はするが、意図がほとんど追えない |
| 3〜5点 | 上級者なら追えるが、かなり難読 |
| 6〜8点 | 中級者でもおおよその流れを追える |
| 9点 | 短く、かつ構造が明快 |
| 10点 | 簡潔さと読みやすさが高いレベルで両立している |

### 減点例

| 問題 | 減点目安 |
|---|---:|
| 過度な記号圧縮 | -2〜-5 |
| 正規表現が説明不能 | -2〜-5 |
| 不要に複雑な `awk` | -2〜-6 |
| コードゴルフ目的の難読化 | -3〜-8 |

---

## F. 堅牢性・安全性: 10点

### 評価観点

入力の揺れ、境界条件、環境差、危険な構文に対してどの程度安全かを評価する。

### LLMが見るべき点

- 空入力に耐えるか
- 空白、タブ、日本語、記号を含む入力に対応できるか
- 最終行に改行がない場合も動くか
- 大量データで極端に遅くならないか
- GNU/BSD差、macOS/Linux差に依存しすぎていないか
- `rm`, `eval`, `xargs`, コマンド置換などを危険に使っていないか
- 未クォート変数による分割やグロブ展開の問題がないか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 危険、破壊的、または環境依存が強すぎる |
| 1〜2点 | 基本入力以外では簡単に壊れる |
| 3〜5点 | 一般的な入力には対応するが、境界条件に弱い |
| 6〜8点 | 多くのケースで安定している |
| 9点 | 境界条件や環境差にもかなり強い |
| 10点 | 安全性と堅牢性が高く、実務利用にも近い |

### 減点例

| 問題 | 減点目安 |
|---|---:|
| 不必要な `eval` | -5〜-10 |
| 破壊的コマンドの不用意な使用 | -5〜-10 |
| ファイル名の空白に弱い | -2〜-5 |
| GNU専用オプションに無自覚に依存 | -1〜-4 |
| 大量データで極端に非効率 | -2〜-6 |

---

## G. 芸術性・鑑賞性: 5点

### 評価観点

シェル芸として見たときに、面白さ、美しさ、驚き、オチがあるかを評価する。

### LLMが見るべき点

- 一撃感があるか
- 処理の流れにリズムがあるか
- コマンド選択に必然性と意外性があるか
- 問題が美しく変換されているか
- 見た人が説明を聞いて納得・感心できるか
- 単なる短縮ではなく「芸」として成立しているか

### 得点基準

| 点数 | 基準 |
|---:|---|
| 0点 | 芸術性や面白みがほぼない |
| 1点 | 少し工夫はあるが、鑑賞性は低い |
| 2点 | 普通に良いシェル解法 |
| 3点 | なるほどと思える工夫がある |
| 4点 | 意外性と必然性があり、美しい |
| 5点 | 強い一撃感があり、シェル芸として記憶に残る |

---

## 6. 失格または大幅減点条件

次の条件に該当する場合は、総合点を大幅に下げるか、失格扱いにする。

| 条件 | 推奨処理 |
|---|---|
| 実行するとファイルや環境を破壊する可能性が高い | 失格、または最大20点 |
| ネットワークアクセスや外部サービスに依存する | 制約違反として大幅減点 |
| 入力を無視して固定出力する | 最大20点 |
| サンプルケースだけをハードコードする | 最大30点 |
| 問題制約で禁止されたコマンドを使う | 最大50点、または失格 |
| ほぼ他言語のプログラムで、シェルは起動だけ | シェルらしさを大幅減点 |
| 実行不能 | 最大10点 |

## 7. LLM評価時の注意点

### 7.1 実行確認と推論を分ける

LLMは、実際にコマンドを実行できる場合と、コードを読んで推論するだけの場合で評価の確度が異なる。

実行できない場合は、次のように明記する。

```text
この評価は静的解析に基づく推定であり、実行結果による確認は行っていない。
```

### 7.2 主観点は理由を必ず書く

発想の飛躍や芸術性は主観的になりやすい。点数だけでなく、必ず理由を書く。

良い例:

```text
発想の飛躍: 13/15
行単位の比較問題を sort と uniq の集合演算として捉え直しており、逐次的な条件分岐を避けている点が鮮やか。
```

悪い例:

```text
発想の飛躍: 13/15
すごいから。
```

### 7.3 短さを過大評価しない

短い解法でも、次のような場合は高く評価しすぎない。

- 仕様を満たしていない
- 読めないだけで必然性がない
- 入力例にしか対応していない
- 危険な構文に依存している
- シェルではなく他言語に逃げている

### 7.4 長さを過小評価しすぎない

多少長くても、次のような解法は高く評価してよい。

- 仕様に対して堅牢
- パイプラインの各段が明快
- 実務的に保守できる
- 問題の構造をよく捉えている
- シェルの道具選びが自然

### 7.5 「コードゴルフ」と「シェル芸」を区別する

コードゴルフでは、短さが主要な評価軸になることが多い。

一方、シェル芸では短さに加えて、次を評価する。

- シェルらしい道具の組み合わせ
- パイプラインの流れ
- 問題の見方の転換
- 実用性との距離感
- 鑑賞性

そのため、最短解が必ずしも最高のシェル芸とは限らない。

## 10. 評価モード別の重み変更

標準モードでは100点の配点を以下のようにする。

| 項目 | 標準 |
|---|---:|
| 正確性・実行可能性 | 30 |
| 簡潔性・無駄のなさ | 15 |
| シェルらしさ | 15 |
| 発想の飛躍 | 15 |
| 可読性・説明可能性 | 10 |
| 堅牢性・安全性 | 10 |
| 芸術性・鑑賞性 | 5 |

### 10.1 競技モード

短さ、発想、一撃感を重視する。

| 項目 | 配点 |
|---|---:|
| 正確性・実行可能性 | 30 |
| 簡潔性・無駄のなさ | 20 |
| シェルらしさ | 15 |
| 発想の飛躍 | 20 |
| 可読性・説明可能性 | 5 |
| 堅牢性・安全性 | 5 |
| 芸術性・鑑賞性 | 5 |

### 10.2 実務モード

保守性、堅牢性、安全性を重視する。

| 項目 | 配点 |

## 11. 最終的な評価思想

シェル芸の中心は、次の問いに集約できる。

> その問題を、シェルの道具立てだからこそ美しく解いているか。

LLMは、評価時に常にこの問いを意識する。

短ければよいわけではない。  
長ければ悪いわけでもない。  
他言語を使えば即失格というわけでもない。  
しかし、シェル芸として高く評価されるには、シェル、パイプ、標準入出力、テキスト処理、既存コマンドの組み合わせに、問題解決としての必然性と美しさが必要である。

## 12. 参考文献と設計上の示唆

このルーブリックは主にシェル芸の実践感覚を言語化したものだが、LLM による自動生成・自動評価を考えるうえでは、関連研究から得られる示唆も有用である。

### 12.1 unix-ctf: Procedural Environments for Unix-Competence Reinforcement Learning

- Source: https://arxiv.org/abs/2605.29115
- 概要:
  この研究は、端末上の課題解決能力を「一般的なプログラミング能力」と「Unix コマンドやシェル機能を道具として使う能力」に分けて扱うべきだと整理している。LLM 支援で生成した課題を用い、Unix 固有スキルを直接訓練・評価する環境を構築している。
- 本ルーブリックへの示唆:
  シェル芸の評価でも、単に正答したかだけではなく、「シェルや UNIX の道具立てを自然に使っているか」を独立した評価軸として持つのが妥当である。特に「シェルらしさ」「発想の飛躍」「CLI ならではの問題分解」を別項目で見る方針を補強する。

### 12.2 Executable Formal Semantics for the POSIX Shell

- Source: https://arxiv.org/abs/1907.05308
- 概要:
  この研究は POSIX shell の実行意味論を機械化し、複数のシェル実装との比較を通じて、シェルの挙動が強力である一方で繊細でもあることを示している。短いコマンドでも、クォート、展開、評価順序、標準入出力の扱いによって結果が大きく変わる。
- 本ルーブリックへの示唆:
  LLM 評価では「短い」「それらしい」だけで高得点にせず、実行可能性、堅牢性、安全性を強く見る必要がある。特に `stderr`、タイムアウト、環境依存、危険な構文、偶然通るだけのワンライナーを減点する根拠として有効である。

### 12.3 補足

- 上記文献はシェル芸の審美眼そのものを定義するものではない。
- ただし、LLM による自動生成・自動評価では、
  「Unix 能力を独立して測ること」
  「シェルの意味論的な難しさを軽視しないこと」
  の 2 点が重要である。
- そのため本ルーブリックでは、正確性だけでなく、簡潔性、シェルらしさ、発想、可読性、堅牢性、安全性、鑑賞性を分離して扱う。
